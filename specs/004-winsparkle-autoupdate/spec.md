# Feature Specification: WinSparkle Auto-Update Key Pair

**Feature Branch**: `004-winsparkle-autoupdate`  
**Created**: 2025-07-10  
**Status**: Draft  

## Summary

WinSparkle requires an Ed25519 key pair to verify appcast update signatures. The current `resources/sparkle_pubkey.txt` is a placeholder. Without a real key pair, auto-update is silently broken — WinSparkle will detect the new version but refuse to install because the signature does not match.

**Scope**: Generate the key pair, embed the public key, update the release pipeline to sign update artifacts, and verify end-to-end update delivery.

---

## User Scenarios & Testing

### User Story 1 - Auto-Update Detected and Verified (Priority: P1)

A user running AirBeam v1.0.0 launches the app; within a few seconds WinSparkle checks the appcast, finds v1.0.1, and offers an upgrade dialog. The user clicks "Install Update" and is upgraded seamlessly.

**Why this priority**: Auto-update is the primary mechanism for delivering security fixes and bug fixes to installed users. A broken auto-update pipeline means users stay on old versions indefinitely.

**Independent Test**: Install a build with the real public key embedded; run `winsparkle-sign` against a test installer to produce a `sparkle:edSignature`; place a crafted appcast pointing to the signed installer; confirm WinSparkle accepts the update.

**Acceptance Scenarios**:

1. **Given** AirBeam v1.0.0 is installed with the Ed25519 public key embedded, **When** WinSparkle polls the appcast URL, **Then** it finds the latest version entry and verifies the `sparkle:edSignature` successfully.
2. **Given** a valid signature, **When** the user clicks "Install Update", **Then** WinSparkle downloads the installer, verifies the signature again, and launches the installer without UAC prompts (silent update path).
3. **Given** an appcast entry with a tampered or missing signature, **When** WinSparkle processes it, **Then** it rejects the update and shows an error — it does NOT install the tampered package.
4. **Given** no internet connectivity, **When** WinSparkle's update check times out, **Then** AirBeam starts normally with no blocking dialog.

---

### User Story 2 - Release Pipeline Signs Artifacts (Priority: P1)

When the release pipeline tags and publishes a new version, the installer is automatically signed with the Ed25519 private key and the `sparkle:edSignature` is written into `appcast.xml`.

**Why this priority**: Manual signing steps are error-prone. If a developer forgets to sign, the auto-update pipeline breaks silently for all existing users.

**Independent Test**: Push a test `v9.9.9` tag; verify the `appcast.xml` generated by the release workflow contains a `sparkle:edSignature` attribute and that `winsparkle-sign --verify` confirms the signature against the known public key.

**Acceptance Scenarios**:

1. **Given** a `v*` tag is pushed, **When** the release workflow completes, **Then** `appcast.xml` contains a `<enclosure>` element with a `sparkle:edSignature` attribute that is non-empty.
2. **Given** the generated `appcast.xml`, **When** `winsparkle-sign --verify <installer> --pubkey <pubkey>` is run in CI, **Then** it exits `0` (valid signature).
3. **Given** the Ed25519 private key is stored as a GitHub Actions secret (`SPARKLE_PRIVATE_KEY`), **When** the release workflow runs in a fork without the secret, **Then** the signing step is skipped with a warning annotation (consistent with the code-signing guard from spec 002).

---

### User Story 3 - Key Pair Rotation (Priority: P3)

If the private key is ever compromised, a new key pair can be generated, the public key updated in the source, and old installers re-signed — without requiring every user to re-install from scratch.

**Why this priority**: Key rotation is an infrequent operation but must be documented so it can be performed correctly under pressure.

**Independent Test**: Generate a second key pair, replace `sparkle_pubkey.txt` with the new public key, rebuild, and verify WinSparkle still verifies updates signed with the new key.

**Acceptance Scenarios**:

1. **Given** a new key pair is generated and the public key is committed to `resources/sparkle_pubkey.txt`, **When** an installer is signed with the new private key, **Then** a build using the new public key accepts the update.
2. **Given** an installer signed with the old private key, **When** a build using the new public key checks it, **Then** WinSparkle rejects the signature (no downgrade attacks via old signed packages).

---

### Edge Cases

- What if `SPARKLE_PRIVATE_KEY` secret is not set in a fork? → Signing step must be guarded with `if: secrets.SPARKLE_PRIVATE_KEY != ''`; appcast is published without signature (update will be detected but not installable — acceptable for fork testing).
- What if the appcast URL is unreachable? → WinSparkle already handles timeouts gracefully; no change needed.
- What if the installer is re-uploaded (e.g., hotfix to the same version)? → The artifact must be re-signed; the `sparkle:edSignature` in appcast must match the new binary.
- How large can the private key secret be? → Ed25519 private key is 32 bytes (64 hex chars) — well within GitHub Actions secret limits.

## Requirements

### Functional Requirements

- **FR-001**: An Ed25519 key pair MUST be generated using `winsparkle-sign --generate-keys` (or equivalent tool that produces WinSparkle-compatible keys).
- **FR-002**: The Ed25519 public key MUST be stored in `resources/sparkle_pubkey.txt` as a Base64-encoded string (WinSparkle format).
- **FR-003**: The Ed25519 private key MUST NEVER be committed to the repository; it MUST be stored only as a GitHub Actions secret (`SPARKLE_PRIVATE_KEY`).
- **FR-004**: `AppController.cpp` (or wherever `WinSparkle_set_dsa_pub_pem` / `WinSparkle_set_ed_dsa_pub_key` is called) MUST pass the public key from `resources/sparkle_pubkey.txt` at startup.
- **FR-005**: The release workflow MUST include a step that calls `winsparkle-sign <installer> --ed-key $SPARKLE_PRIVATE_KEY` and captures the resulting Base64 signature.
- **FR-006**: The appcast generation step MUST embed the captured signature as `sparkle:edSignature` on the `<enclosure>` element.
- **FR-007**: The signing step MUST be guarded: `if: secrets.SPARKLE_PRIVATE_KEY != ''`.
- **FR-008**: When the signing step is skipped, a `::warning::` annotation MUST be emitted so the release author is aware the update is unsigned.
- **FR-009**: `AIRBEAM_APPCAST_URL` in `CMakeLists.txt` MUST be updated from `TODO_ORG` to the real GitHub Pages URL before the first release. The repo's final GitHub location (org vs personal account) is TBD; this is a pre-release prerequisite documented in the release checklist.
- **FR-010**: Key generation instructions MUST be documented in `docs/release-process.md` (or equivalent) for key rotation procedures.

### Key Entities

- **`resources/sparkle_pubkey.txt`**: Base64-encoded Ed25519 public key; committed to repo; embedded in the binary at build time.
- **`SPARKLE_PRIVATE_KEY`** (GitHub secret): Ed25519 private key; used only in the release signing step; never stored in source.
- **`appcast.xml`**: WinSparkle-compatible appcast; must contain `sparkle:edSignature` on the installer `<enclosure>` element.
- **`winsparkle-sign`**: CLI tool (part of WinSparkle distribution) used for key generation and signing.

---

## Success Criteria

### Measurable Outcomes

- **SC-001**: `winsparkle-sign --verify <installer> --pubkey resources/sparkle_pubkey.txt` exits `0` for any installer produced by the release pipeline.
- **SC-002**: A WinSparkle-enabled build detecting a signed appcast entry proceeds through "Install Update" without a certificate error dialog.
- **SC-003**: An appcast entry with a deliberately corrupted signature causes WinSparkle to abort the update and display an error.
- **SC-004**: The release pipeline completes without manual intervention when `SPARKLE_PRIVATE_KEY` is set in the repo secrets.
- **SC-005**: `resources/sparkle_pubkey.txt` contains a non-placeholder Ed25519 public key (length ≥ 43 Base64 characters).
